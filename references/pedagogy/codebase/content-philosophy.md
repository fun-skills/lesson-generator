# 内容理念（代码库课程差异教学法）

> 何时阅读：写 codebase 课程时阅读。共同教学法见 `pedagogy/*.md`，本文件只放代码库课程的差异规则。普通课程不读本文件。

代码库课程面向 vibe coder：他们用自然语言指挥 AI 编码工具，但没有系统读过代码。目标是看懂项目结构、更准确地指挥 AI、发现 AI 把逻辑放错地方、在 AI 卡住时知道从哪里查、和工程师讨论架构时不迷路、学会能写进 prompt 的技术词汇。

## 展示，而非讲述

课程应该像信息图，而不是教科书：

- 每节至少一半内容应该是视觉或互动 block。
- 3 个以上项目不要写成长列表，改成 `list-block`。
- 解释代码不要写段落，改成 `code-translation`。
- 解释组件通信不要写段落，改成 `actor-chat` 或 `flow`。
- 解释文件职责不要写段落，改成 `architecture` 或 `debug-case`。

## 代码 ↔ 自然语言解释

最重要的教学元素。每节课至少一个 `code-translation`，左侧真实代码，右侧白话中文解释。

- 代码必须来自目标代码库，不改写、不清理、不伪造。
- 太长就选自然短小的片段，不要改写。
- 解释说"为什么这么写"，不只说"它是什么"，并帮助学习者以后更准确地指挥 AI。

## 先隐喻，后代码

用日常隐喻建立直觉，然后立刻落回真实代码。不要重复同一个隐喻，不要默认用"餐厅"解释所有东西。

## 通过追踪来学习

从用户已经见过的动作开始追踪：

```text
你点击这个按钮后，数据先进入 A，再由 B 处理，最后 C 更新界面。
```

第一节从一个具体用户操作开始，再追踪背后的代码。

## 给组件角色感

主要组件不是图上的方框，而是故事里的角色。用 `actor-chat` 让它们说话：前端问后端要数据、后端问数据库查记录、缓存提醒"我已经有这个结果"。

## 术语必须解释

技术术语第一次出现时都要解释：API、JSON、DOM、CLI、SDK、缓存、函数、变量、模块、组件、状态、数据库、队列、认证、中间件、构建、打包、入口点、环境变量，以及学习者可能不知道的软件或服务名称。词汇本身就是课程目标之一。

## 测验考应用，不考记忆

优先题型：调试场景（先查哪里）、架构决策（放哪一层）、追踪练习（数据路径）、AI 指挥（怎么提示 AI 改）。避免"API 是什么缩写""哪个文件处理 X"这类记忆题。

## 每节课的最低内容

每节课至少：`goal` / 2-4 个核心内容 block / 一个真实代码或架构相关 block / `flashcards` / `quiz` / `sources`。

整门课至少出现：`code-translation` / `actor-chat` / `flow` / `architecture` 或 `debug-case`。

## 推荐学习曲线

| 位置 | 目的 |
|---|---|
| 1 | 项目做什么，一次真实操作如何发生 |
| 2 | 认识主要角色 |
| 3 | 数据怎么流动 |
| 4 | 外部世界、依赖、成本和失败点 |
| 5 | 关键模式，学会给 AI 更精确的指令 |
| 6 | 出错时怎么查 |
| 总复习 | 综合 |

根据代码库调整，不要机械套模板。
