# 互动元素（代码库课程差异教学法）

> 何时阅读：写 codebase 课程的 `body` 时阅读。block 结构见 `references/schema.md`，本文件讲代码相关 block 怎么用。

## 总规则

- 只写 block 数据，不写 DOM、内联 `<style>`、内联 `<script>`。
- 当前 runtime 不支持的互动，先不要临时塞 HTML，记录为后续扩展。

## code-translation

真实代码与白话中文解释。

- `code` 必须是真实代码。
- `file` 写文件路径，推荐带行号。
- `explanation` 是数组，每项解释 1-2 行代码，贴近代码中的一小段。
- 适合：把一段真实代码讲给非技术学习者。

## actor-chat

把组件通信讲成对话。适合：前端和后端通信、控制器和服务通信、Agent 和工具通信、缓存/数据库/队列等角色协作。

- 角色名来自真实代码角色。
- 对话推动理解，不伪造模块职责。

## flow

展示数据如何一步步移动。

- `steps[].from` 和 `steps[].to` 必须来自 `nodes`，否则运行时无法正确高亮。
- 每一步只讲一个动作。
- 文案要让非技术学习者能追上。
- 对应真实用户路径或数据路径。

## architecture

展示主要角色和依赖关系。适合：课程第 2 节认识角色、总复习、解释新功能应该放在哪里。

- 角色来自真实文件、模块、组件或服务。
- 边的含义具体，不画和源码不一致的架构。

## debug-case

建立调试直觉。适合：用户反馈功能坏了、数据不刷新、API 失败、构建失败、AI 反复改错地方。

- 症状像真实用户反馈。
- `firstFiles` 必须是真实文件。
- `fixHint` 指导排查方向，不编造修复。

## ai-dialog

展示学习者如何和 AI 沟通，教学习者写更好的 prompt。

- prompt 要具体到文件、角色、数据流或失败症状。
- 不写泛泛的"帮我优化代码"。
